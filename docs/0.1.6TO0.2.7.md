# Architecture Evolution: v0.1.6 -> v0.2.7

This document explains the major architecture changes introduced between `v0.1.6` and `v0.2.7`, with a focus on `packages/web`.

## Why this rewrite happened

The `0.2.x` line moves iKanban from a mostly bundled "web + custom backend runtime" shape into a cleaner "web client + OpenCode server" architecture.

At a high level, this gives us:

- clearer separation of responsibilities between UI and backend,
- better compatibility with OpenCode SDK v2 event/data contracts,
- more predictable routing for local and GitHub Pages deployments,
- easier testing and evolution of frontend state layers.

## Core architecture position in `0.2.x`

We removed the old custom backend that lived inside `packages/web` and now treat iKanban as a web client first.

- iKanban is local-first in terms of UI/runtime ownership.
- OpenCode provides backend capabilities through its API and event stream.
- Browser capabilities (storage, notifications, routing, cross-device access) are combined with OpenCode APIs to implement product features.

This is simpler and more portable, but it also introduces constraints and risks that are important to call out.

## 1) Runtime boundary changed (biggest structural shift)

### Before (`v0.1.6`)

- `packages/web` shipped an in-package Node/Express backend (`packages/web/server/index.js`) with a large set of custom routes and integrations.
- The package entry pointed at that server (`"main": "./server/index.js"` in `packages/web/package.json` at `v0.1.6`).
- The browser app depended on a runtime API bridge (`packages/web/src/api/*`) and injected `window.__IKANBAN_RUNTIME_APIS__` from `packages/web/src/main.tsx`.

### After (`v0.2.7`)

- `packages/web` is primarily a frontend app/library (`packages/web/src/*`) and exports app primitives from `packages/web/src/index.ts`.
- The old in-package server tree is removed from the package runtime path.
- The CLI (`packages/web/bin/cli.js`) is intentionally small: static file serving + proxying OpenCode API prefixes to `OPENCODE_URL`.

### Impact

- less duplicated backend logic in iKanban,
- fewer moving parts inside `packages/web`,
- easier alignment with OpenCode server behavior.

### Risk and tradeoff

Because we removed our own backend layer, feature velocity is now more tightly coupled to OpenCode-exposed APIs.

- If OpenCode does not expose a capability, we cannot implement it purely in our web layer.
- Some behaviors that previously could be implemented server-side in iKanban now need upstream API support.
- Our extension surface is therefore "API-first", not "arbitrary backend logic".

## 2) Frontend stack changed to a provider-driven Solid architecture

`v0.2.7` builds a structured provider tree in `packages/web/src/app.tsx`:

- app-scoped providers (`SettingsProvider`, `LayoutProvider`, `NotificationProvider`, etc.),
- server and SDK providers (`ServerProvider`, `GlobalSDKProvider`, `GlobalSyncProvider`),
- route-scoped session providers (`TerminalProvider`, `FileProvider`, `PromptProvider`, `CommentsProvider`).

This layered composition replaces the older runtime-API-injection pattern and makes boundaries explicit:

- global server/session graph state,
- per-directory data sync,
- per-session UI interaction state.

## 3) Data synchronization was redesigned around SDK event streams

Two contexts now define the data flow contract:

### `GlobalSDK` (`packages/web/src/context/global-sdk.tsx`)

- opens a long-lived global SSE stream,
- batches/coalesces noisy event types (`session.status`, `message.part.updated`, `lsp.updated`),
- applies heartbeat + reconnect behavior for reliability,
- emits events by directory namespace.

### `GlobalSync` (`packages/web/src/context/global-sync.tsx`)

- keeps global state (`project`, `provider`, `config`, `path`),
- manages per-directory child stores with lifecycle/eviction,
- bootstraps global and directory state separately,
- routes events through reducers (`applyGlobalEvent`, `applyDirectoryEvent`),
- caches project metadata and session todo snapshots.

Compared to `0.1.6`, this is a major shift from ad-hoc request-driven runtime APIs to an event-first synchronization layer.

## 4) Static web deployment is now a strategic advantage

The browser-first runtime allows straightforward static deployment while still connecting to OpenCode.

- deploy once and access from many devices,
- use over public network or local area network,
- keep a consistent experience across laptops/desktops/tablets without shipping a full custom backend.

In practice, this significantly improves usability for multi-device workflows.

## 4.1) New `0.2.x` usage model (no iKanban backend boot required)

In `0.2.x`, the default onboarding flow is much simpler:

1. Open the hosted static app: `https://isomoes.github.io/ikanban`
2. Start your OpenCode server (for GitHub Pages usage):

```bash
opencode serve --port <PORT> --cors https://isomoes.github.io
```

3. In iKanban settings, connect your custom OpenCode backend URL (for example: `http://localhost:<PORT>`).

This means users do not need to start a separate iKanban backend service just to use the web UI.
The static page + configurable OpenCode backend is now the primary usage path.

For exact command examples and alternatives (including local `npx` mode), see `README.md`.

## 5) Multi-project and workspace model expanded

`0.2.x` introduces stronger multi-project behavior:

- multiple projects can stay open at the same time,
- workspace-level organization and switching are first-class,
- git worktree-based parallel flows are supported, making it easier to run parallel changes in one repo.

This is a foundational productivity upgrade for users working on more than one branch/task concurrently.

## 6) UI architecture changed, especially review workflows

The review panel experience was redesigned.

- In earlier versions, review was more isolated as a separate panel workflow.
- In `0.2.x`, review is integrated with per-file changed content views, improving continuity with session and file navigation.

### Current limitation

Review content depends on modified content provided by OpenCode backend APIs. It is not yet a full real-time, global local-filesystem diff engine.

This is an explicit area we plan to keep improving.

## 7) What this means for contributors

When adding features in the `0.2.x` architecture:

1. Treat OpenCode as the backend source of truth, not `packages/web` server code.
2. Put cross-project/session data in `GlobalSync` and per-directory/session behavior in scoped contexts.
3. Use event reducers and optimistic sync hooks (`sync.tsx`) instead of one-off imperative refreshes.
4. Keep URL directory/session semantics intact; avoid hidden global active-project assumptions.
