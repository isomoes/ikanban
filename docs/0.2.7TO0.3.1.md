# Product Evolution: v0.2.7 -> v0.3.1

This document introduces the major changes shipped between `v0.2.7` and `v0.3.1`.
It focuses on the product direction of the `0.2.8` to `0.3.1` release line: turning iKanban from a capable OpenCode web client into a more complete multi-agent workspace with a clearer home view, stronger review flow, and a session-centered task graph.

## Snapshot

Between `v0.2.7` and `v0.3.1`, iKanban changed in three visible ways:

- the home screen became a real session status board instead of just a navigation surface,
- review moved closer to the active session workflow and became easier to read and navigate,
- session pages gained an ikanban task graph that makes parent and child agent work easier to inspect.

This release line also cleaned up surrounding UX details such as routing, session switching, settings behavior, server selection, documentation, and release packaging.

## UI progression

### Earlier direction

`v0.2.7` already had the browser-first architecture from the `0.2.x` migration, but the product still leaned more heavily on chat/session primitives than on a full kanban-style workspace view.

### Current direction

By `v0.3.1`, the app is much more explicit about the main workflow:

- start from a home board,
- jump into an active session,
- review changes in-place,
- inspect child-agent structure through the session graph,
- return to the board to track broader progress.

That flow is the real upgrade in this version range.

## 1. Home became a real coordination surface

The home experience changed a lot during this period.

- `v0.2.10` introduced a session status board home view.
- `v0.2.11` simplified home/session routing, hid the shell on home, and added a faster return path.
- `v0.2.12` skipped unnecessary project bootstrap work on home so the landing page became lighter.
- `v0.2.13` narrowed the board to opened sessions so it focused on active work.
- `v0.3.1` excluded child sessions from the idle board so parent-agent overviews no longer show duplicated cards.

The result is a much better starting point for multi-agent work. Home is no longer just where you land before opening a session. It now acts as the board-level overview for active work in the current project.

## 2. Review became more central and more readable

This release line significantly improved review workflows.

- `v0.2.8` added project-wide changes review alongside session diffs, plus timeline jump controls and file review toggles.
- `v0.2.9` added persistent word wrap controls, word-level diff highlighting, and better GitHub Dark Colorblind diff colors.
- `v0.2.14` moved diff context into file tabs, removed the separate file tree review flow, and restored renderable diff context.

This matters because iKanban is not just for prompting agents. It is also for checking their work quickly. By the end of this release range, review is more integrated into the main session surface and asks for fewer mode switches.

## 3. Session pages gained an ikanban task graph

The biggest new surface in `v0.3.0` is the session ikanban task graph.

- sessions can now expose parent/child task structure more visually,
- node details moved into the main pane instead of living in a detached detail view,
- archive state is stored in browser storage so graph organization survives reloads,
- session tab borders were restored to keep session boundaries clearer while navigating.

This is the clearest expression so far of the product name: iKanban is not only a chat wrapper around OpenCode sessions. It is becoming a task-oriented, graph-aware workspace for supervising agent execution.

## 4. Navigation and settings became less noisy

Several smaller changes made daily usage smoother:

- the command palette shortcut became global in `v0.2.11`,
- server picker row actions became more predictable in `v0.2.12`,
- the desktop sidebar toggle was removed in `v0.2.12` to reduce duplicate navigation controls,
- update prompts were disabled in `v0.2.8`, while changelog links were surfaced from settings,
- built-in language support stayed focused on English and Chinese.

Individually these are small, but together they make the app feel more deliberate and less cluttered.

## 5. Packaging and project structure were simplified

There were also important maintenance changes behind the scenes.

- `v0.2.14` vendored the remaining shared UI code and removed obsolete workspace packages.
- `v0.2.15` replaced broken public asset symlinks with tracked files so release and GitHub Pages builds became more reliable.
- SDK versions were updated in `v0.2.10` and `v0.2.14` to stay aligned with upstream changes.

These changes reduce release friction and make the web package easier to ship and maintain.

## 6. Documentation now matches the hosted and multi-agent workflow better

The docs also moved forward during this range.

- the README was refreshed multiple times,
- the multi-agent intro was clarified,
- a release prompt checklist was added,
- the project now includes stronger visual documentation in the README.

This is important because the product is now opinionated about a hosted UI plus configurable OpenCode backend flow, and the docs increasingly reflect that reality.

## What changed in practice

For users upgrading from `v0.2.7`, the practical difference is straightforward:

- you get a much better at-a-glance home board,
- you can review agent output with less friction,
- you can inspect multi-agent task structure directly inside the session page,
- the app feels more focused on coordination and supervision rather than only conversation.
